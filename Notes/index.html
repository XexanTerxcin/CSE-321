04.1

Network Layer Protocols

This is the layer whose entire job is: figure out where this data is going, and pick the best road to get it there.

  • IP (Internet Protocol)→Gives every device an address and figures out routing. Basically the postal system of the internet — every packet gets a "from" and "to" address stamped on it.
  • Internetworking→Stitching different networks together so they behave like one giant network. The Internet is literally "a network of networks."

Routing Protocols — how a router decides "which way do I send this?"

RIP

Routing Information Protocol

Distance vector. Just counts how many "hops" (routers) are between here and the destination, and picks the path with the fewest.

Like asking a stranger for directions who only knows "it's 3 streets that way" — no idea about traffic or shortcuts.
OSPF

Open Shortest Path First

Link state. Every router builds a full map of the entire network, then calculates the actual shortest path. Faster to adapt when something changes.

Like having GPS with the complete map already loaded, recalculating instantly if a road closes.
BGP

Border Gateway Protocol

Path vector. Used between huge networks — this is the protocol that literally runs the internet's backbone, connecting ISPs to each other.

Like different countries negotiating which highways connect at their borders.
04.2

Transport Layer Protocols

Once data reaches the right device, the Transport Layer decides how carefully it gets delivered. Two philosophies: be careful, or be fast. Hit run and watch the difference.

TCPreliable · ordered · double-checks everything
UDPfast · no guarantees · doesn't look back

Notice: UDP finishes first every time — but if a packet got lost along the way, UDP simply never sends it again. TCP arrives a little slower because it keeps stopping to confirm "did that arrive okay?" before moving on.

TCP

Transmission Control Protocol

Connection-oriented and reliable. Sets up a handshake first, numbers every packet, waits for acknowledgments, and resends anything lost.

Like registered mail — you get a tracking number and a signature confirming it arrived.
UDP

User Datagram Protocol

Connectionless and fast. No handshake, no acknowledgment, no resending. It just fires the data and moves on.

Like tossing postcards in a mailbox — quick, but nobody guarantees they show up.

Flow Control & Error Correction (this layer's version)

  • Flow control→The sender paces itself so it doesn't flood the receiver — same sliding-window idea as the Data Link Layer, just now happening end-to-end across the whole journey.
  • Error correction→TCP attaches a checksum to every segment. If something looks damaged or never gets acknowledged, it just resends it.

04.4

Internet Protocol Suite — the support crew

These three don't carry your actual data. They quietly keep the whole system running behind the scenes.

ICMP

Internet Control Message Protocol

Used for error reporting and diagnostics — this is exactly what powers ping and traceroute.

Like a courier calling back to say "this address doesn't exist" instead of silently disappearing.
DNS

Domain Name System

Translates domain names into IP addresses. You type google.com, it quietly hands your device the real numeric address behind it.

Like the internet's phonebook — you look up a name, it gives you a number.
DHCP

Dynamic Host Configuration Protocol

Automatically hands your device an IP address the moment it joins a network — no manual setup needed.

Like checking into a hotel and being handed a room number at the front desk, instead of having to claim one yourself.
04.5

Advanced Topics

A closer look at four areas that build on everything above — how networks stay safe, how they go wireless, how they handle live video/audio, and what you're actually using all day.

Security is basically: how do we make sure only the right people can see or touch this data? A few layers work together to pull that off.

  • Encryption→Scrambles your data into gibberish so that even if someone intercepts it, they can't read it without the key. It's like locking a message in a box that only the recipient has the key for.
  • Firewalls→A gatekeeper that inspects traffic entering or leaving a network and blocks anything that breaks the rules — like a bouncer checking IDs at the door.
  • VPN (Virtual Private Network)→Creates an encrypted "tunnel" through the public internet. Like riding in a blacked-out car through a busy city — people see the car pass by, but not what's inside.
  • Authentication→Proving you actually are who you claim to be — passwords, fingerprints, 2FA codes, etc.
Common threats worth knowing
  • Malware — malicious software that sneaks onto a device to steal data or cause damage.
  • Phishing — fake emails/messages tricking you into giving up passwords or info.
  • DDoS attack — flooding a server with junk traffic from thousands of sources until it collapses under the load.
  • Man-in-the-middle — someone secretly sits between you and the server, eavesdropping or altering data in transit.
Wi-Fi

IEEE 802.11

Local wireless network. A router broadcasts a signal (2.4GHz or 5GHz radio waves), and devices connect using an SSID (network name) and password.

Range: a home, office, or campus — short-to-medium distance, but fast.
Cellular

3G / 4G / 5G

Wide-area mobile network. Your phone connects to the nearest cell tower, and as you move, it hands you off to the next tower without dropping the call.

Range: entire cities and countries — each generation gets faster and lower-latency.

Why wireless is trickier than wired

No physical cable means the signal is exposed to interference (walls, microwaves, other networks), limited range (signal weakens with distance), and extra security risk (anyone nearby can technically "hear" the signal, which is exactly why Wi-Fi passwords and encryption matter so much).

Streaming video and making a video call are way more demanding than loading a webpage — the network needs to keep up in real time, not just eventually deliver everything.

  • QoS (Quality of Service)→A set of guarantees about bandwidth, delay, and reliability — basically the network promising "I'll prioritize this traffic so your call doesn't freeze."
  • Latency→The delay before data arrives. Too much latency = that awkward lag on a video call.
  • Jitter→Inconsistent delay — sometimes fast, sometimes slow. This is what makes audio choppy even if average speed is fine.
  • Buffering→Loading a small chunk ahead of time so brief network hiccups don't interrupt playback.
  • Adaptive bitrate streaming→Netflix/YouTube automatically lower your video quality when your connection gets weak, instead of just freezing.
Key idea: for a video call, it's better to briefly drop a video frame than to pause and wait for it — live media cares more about staying on time than being complete. That's actually one reason real-time video/voice often rides on UDP instead of TCP.
Client–Server model

The request/response pattern

A client (your browser, your email app) sends a request, and a server sends back a response. Almost everything you use daily works this way.

Examples: web browsing (HTTP/HTTPS), email (SMTP/IMAP/POP3), cloud storage.
Peer-to-Peer model

No middleman server

Devices talk directly to each other and share the workload, instead of everyone depending on one central server.

Examples: file-sharing networks, some video call setups.

Every app you touch — browser, email, video calls, IoT gadgets — is really just this whole stack (from Module 1's cables up through today's protocols) working together underneath a friendly interface.

Design, Configuration & Troubleshooting

Everything so far was "how a network works." This module is "how someone actually builds, sets up, and fixes one."

05.1

Network Design Principles

Four things every good network is designed around — think of it like designing a city's road system.

01

Scalability

The network can grow — more users, more devices, more traffic — without needing a total rebuild.

02

Security

Protects against unauthorized access, attacks, and data leaks.

03

Redundancy

Backup paths and devices, so if one part fails, the network doesn't go down with it.

04

Cost-effectiveness

Balances performance needs against the actual budget — no point overbuilding what nobody needs.

05.2

Network Configuration

The physical (and virtual) gear that actually makes a network run, and what job each piece does.

Router

Connects different networks

Decides how traffic moves between separate networks — the device making routing decisions from Module 4 in real life.

Switch

Connects devices within one network

Forwards data to the right device using MAC addresses, so traffic doesn't get blasted to everyone.

Firewall

The security gate

Filters traffic in and out based on rules, blocking anything suspicious before it reaches the network.

Management tools

Eyes on the whole network

Software (like SNMP-based dashboards) for monitoring devices remotely, watching traffic patterns, and catching problems early — before users even notice.

05.4

Case Studies

This is the "put it all together" part — taking everything from Modules 1 through 5 and applying it to a real, messy situation.

SCENARIO — "The office Wi-Fi is unbearably slow at 2pm every day"

Step 1 — Symptom: Speed drops specifically during a known time window, not randomly.

Step 2 — Isolate: Everyone's affected, not just one device → points to something shared (the router, the bandwidth, or interference), not one broken cable.

Step 3 — Test: Check device count on the network at 2pm (lunch break = everyone's phone joins Wi-Fi), and run a monitoring tool to see bandwidth usage.

Step 4 — Fix: Turns out too many devices are competing for one access point → the real fix is a design problem (scalability), not a broken part. Add a second access point or upgrade bandwidth.

Step 5 — Verify: Re-test at 2pm the next day.

This is exactly the kind of scenario Module 5 case studies train you to reason through — connecting topology, protocols, and design principles to a practical fix instead of guessing.
